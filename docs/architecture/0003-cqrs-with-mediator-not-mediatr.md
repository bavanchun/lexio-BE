# 0003. CQRS with Mediator (not MediatR)

Date: 2026-05-10

## Status

Accepted

## Context

CQRS separates read and write paths, simplifying handler logic and enabling independent scaling.
The de-facto .NET CQRS library is **MediatR** (Jimmy Bogard). However:

- MediatR became commercially licensed (dual MIT/commercial) from v13 onward for teams using it
  in a commercial product. The exact terms are subject to change.
- **martinothamar/Mediator** (`Mediator` NuGet) is MIT-licensed, source-generator-based
  (zero reflection at runtime), and produces near-zero overhead dispatch.

## Decision

Use **martinothamar/Mediator** (`Mediator.SourceGenerator` + `Mediator.Abstractions`, both MIT)
for in-process command/query dispatch within each service's Application layer.

Interfaces used:
- `ICommand<TResponse>` / `ICommandHandler<TCommand, TResponse>`
- `IQuery<TResponse>` / `IQueryHandler<TQuery, TResponse>`
- `IPipelineBehavior<TMessage, TResponse>` for cross-cutting behaviours (logging, validation)

`IMediator` is injected by the source generator's `AddMediator()` DI extension registered in
each service's `Application.DependencyInjection`.

## Consequences

**Positive:**
- MIT license; no commercial concern for production use.
- Source-gen dispatch: no reflection, near-zero allocations vs MediatR's dynamic invocation.
- Familiar CQRS pattern; any MediatR developer adapts in minutes.

**Negative:**
- Smaller community than MediatR; fewer third-party pipeline behaviours available out-of-the-box.
- Source generator requires `Mediator.SourceGenerator` package in the **Api** project (not Application)
  to keep Application free of generator dependencies — a subtle split.

**Neutral:**
- FluentValidation pipeline behaviour is wired manually in `ValidationBehavior.cs` (generated by template).
- If MediatR's licensing is clarified favorably, migration is straightforward (same pattern).
